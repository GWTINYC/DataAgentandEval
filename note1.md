正在进行一项知识库搭建的任务，最终要搭建一个智能问答助手，帮助业务同学理解业务代码，提升回答的准确度和质量，而且要求业务视角的语言风格，不能是死板僵硬的代码逻辑。pipeline分3层：数据层、编译层和生成层。数据层是把原始代码仓库变成一个个业务域的业务视角的知识卡片；编译层是把一个个知识卡片串起来，形成知识图谱或者叫LLM Wiki（借鉴卡帕西的思想）；生成层是针对用户（业务同学）的问题，根据编译的Wiki进行召回并生成出业务视角的回答，体现对代码的融会贯通的高级理解，帮助业务同学理解代码。
现在我在开发这个知识库的时候，遇到了一些问题，为我提供解决思路。1. 大问题是：如何对于这个流水线（Agent）进行迭代优化，我们目前能拿到业务同学对于一些典型问题的标注答案（黄金数据集），但是数量有限。而且我理解这个黄金数据集是与生成层关系最密切，因为标注的是最后生成的理想答案，与前面两层关系不太密切，所以如何对整条链路进行评测、迭代、优化呢？2. 小问题是：关于数据层，我们目前采取的策略是二阶段处理，先从原始代码仓 to json格式的层次化的代码描述，json包含了粗粒度的代码层级和联系；再从json to .md格式的知识卡片，希望能够深度挖掘json体现出的代码之间的深层次的联系，进行更精细化的描述。因为json包含层级，比原始代码仓更有逻辑，应该更容易转化为知识卡片，所以采用二阶段处理策略，希望能生成的知识卡片质量更高。小问题是这样的处理合理吗？如何对数据层进行评测、迭代、优化呢？

## 先给结论

你的判断只对了一半：**黄金答案确实最直接评测生成层，但它可以反向拆解成前两层的评测标签**。不要只把黄金数据集理解成“最终回答长什么样”，而要扩展成：

> 用户问题 → 理想回答 → 必须覆盖的业务概念 → 必须命中的知识卡片 → 必须经过的图谱关系路径 → 必须引用的代码证据 → 禁止胡说的边界

这样一来，黄金数据集就能评测整条链路，而不是只评测最后的生成层。

# 1. 整条 pipeline 怎么评测、迭代、优化？

你们的三层可以对应成一个“代码理解编译器”：

| 层级  | 作用             | 核心评测问题                  |
| --- | -------------- | ----------------------- |
| 数据层 | 原始代码 → 业务知识卡片  | 卡片有没有把代码讲成业务？有没有遗漏关键逻辑？ |
| 编译层 | 卡片 → Wiki / 图谱 | 卡片之间的关系是否正确？能否支持跨模块理解？  |
| 生成层 | 问题 → 召回 → 回答   | 回答是否准确、有证据、业务同学看得懂？     |

RAG/Agent 评测不能只看最终答案。LangSmith 的 RAG 评测流程也是先建立测试集，再运行应用，最后分别评估答案相关性、答案准确性和检索质量；Ragas 也把 RAG 指标拆成 Context Precision、Context Recall、Faithfulness、Response Relevancy 等维度。([LangChain Docs][1]) ([Ragas][2])

## 1.1 把黄金数据集升级成“多层标注集”

你们现在可能是这样：

```json
{
  "question": "订单取消后，退款流程是怎么走的？",
  "golden_answer": "当订单取消后，系统会先判断订单状态..."
}
```

建议升级成这样：

```json
{
  "question": "订单取消后，退款流程是怎么走的？",

  "golden_answer": "业务视角的理想答案",

  "expected_business_concepts": [
    "订单取消",
    "退款发起",
    "退款状态流转",
    "支付渠道回调",
    "异常补偿"
  ],

  "expected_cards": [
    "order_cancel_flow.md",
    "refund_service.md",
    "payment_callback.md",
    "refund_compensation.md"
  ],

  "expected_graph_paths": [
    "订单取消 -> 触发退款申请 -> 调用支付退款 -> 等待支付渠道回调 -> 更新退款状态",
    "退款失败 -> 进入补偿任务 -> 重试或人工处理"
  ],

  "expected_code_evidence": [
    "OrderCancelService.cancelOrder",
    "RefundService.createRefund",
    "PaymentCallbackHandler.handleRefundCallback",
    "RefundRetryJob"
  ],

  "must_have_points": [
    "说明什么场景会触发退款",
    "说明退款不是立即完成，而是依赖支付渠道回调",
    "说明异常情况下有补偿机制"
  ],

  "forbidden_claims": [
    "不能说取消订单一定立即到账",
    "不能遗漏支付渠道回调",
    "不能把退款和退货混为一谈"
  ],

  "style_rubric": {
    "business_language": true,
    "avoid_code_jargon": true,
    "use_process_explanation": true
  }
}
```

这样一个样本可以同时评测：

* **数据层**：有没有生成这些知识卡片？卡片里有没有这些业务概念？
* **编译层**：图谱里有没有这些关系路径？
* **召回层**：用户问这个问题时，能不能召回这些卡片？
* **生成层**：最终答案有没有覆盖 must-have points，是否出现 forbidden claims？

这就是关键：**把最终答案拆成中间可验证的“业务事实链”。**

---

## 1.2 建立四级评测，而不是一个总分

建议你们不要只打一个“回答准确率”。应该拆成四层分数。

### 第一级：数据层 Card Eval

评测知识卡片本身。

| 指标                | 含义                | 示例                          |
| ----------------- | ----------------- | --------------------------- |
| Coverage 覆盖率      | 关键业务逻辑有没有被卡片覆盖    | 退款、取消、补偿、回调是否都有卡片           |
| Groundedness 可追溯性 | 卡片里的每个关键判断能否追溯到代码 | 每个业务结论绑定 file/function/line |
| Abstraction 抽象质量  | 是否从代码函数提升成业务语言    | 不是“调用 xxx 方法”，而是“发起退款申请”    |
| Completeness 完整性  | 是否讲清主流程、异常流、边界条件  | 成功退款、退款失败、重复回调              |
| Freshness 新鲜度     | 代码变更后卡片是否更新       | commit 变化后重新生成相关卡片          |

Google 的 Code Wiki 也强调“持续更新的结构化 wiki”，并且 wiki 内容需要链接到相关代码文件和定义；这对你们的数据层设计很有参考价值。([developers.googleblog.com][3])

---

### 第二级：编译层 Wiki / Graph Eval

评测卡片之间是否形成了真正的“业务理解网络”。

| 指标                        | 含义          | 示例                  |
| ------------------------- | ----------- | ------------------- |
| Entity Accuracy           | 业务实体是否正确    | 订单、退款单、支付单、售后单有没有混淆 |
| Relation Accuracy         | 关系是否正确      | 订单取消是否真的触发退款申请      |
| Path Recall               | 关键业务链路是否能走通 | 从“订单取消”能否走到“支付回调”   |
| Cross-domain Connectivity | 跨域连接是否完整    | 订单域、支付域、售后域是否连起来    |
| Redundancy                | 是否大量重复卡片    | 多张卡片重复解释同一流程        |

GraphRAG 的核心价值就在于解决普通向量 RAG “connect the dots” 能力弱的问题，尤其适合需要跨多个信息片段综合理解的场景。([微软 GitHub][4])
你们的代码业务问答正是这种场景：用户问的不是单个函数，而是“一个业务动作背后跨模块发生了什么”。

---

### 第三级：召回 Eval

评测用户问题来了之后，系统有没有拿到正确上下文。

| 指标                | 含义                |
| ----------------- | ----------------- |
| Card Hit@K        | Top K 结果里是否包含关键卡片 |
| Evidence Recall   | 是否召回了关键代码证据       |
| Graph Path Recall | 是否召回了完整业务链路       |
| Context Precision | 召回内容是否少而准         |
| Context Recall    | 关键上下文是否没有漏        |

Ragas 里 Context Precision、Context Recall、Context Entities Recall 这些指标正好可以借鉴，用来判断召回上下文是否有用、是否覆盖问题所需实体。([Ragas][2])

---

### 第四级：生成 Eval

评测最终答案。

| 指标                 | 含义               |
| ------------------ | ---------------- |
| Answer Correctness | 和黄金答案是否一致        |
| Faithfulness       | 是否严格基于召回内容回答     |
| Business Style     | 是否像业务语言，而不是代码注释  |
| Completeness       | 是否覆盖主流程、异常流、边界条件 |
| Actionability      | 业务同学看完是否知道系统怎么运转 |
| Refusal Quality    | 证据不足时是否明确说“不确定”  |

LangSmith 文档里也提到，RAG 评测可以分为需要参考答案的评测、不需要参考答案的评测，以及 pairwise 对比评测；这很适合你们做“版本 A vs 版本 B”的效果比较。([LangChain Docs][5])

---

## 1.3 错误归因：每次坏答案都要定位是哪一层错

你们需要给每一次回答保存 trace：

```text
用户问题
 ↓
意图识别结果
 ↓
召回到的知识卡片
 ↓
召回到的图谱路径
 ↓
召回到的代码证据
 ↓
生成模型的中间 reasoning 摘要
 ↓
最终回答
 ↓
评测结果
```

然后用下面的归因规则：

| 现象             | 主要问题层                 |
| -------------- | --------------------- |
| 正确知识卡片根本不存在    | 数据层问题                 |
| 卡片存在，但关系没连起来   | 编译层问题                 |
| 关系也存在，但没召回     | 召回策略问题                |
| 召回到了，但回答没用上    | 生成层问题                 |
| 回答用了，但表达太技术化   | 生成层 prompt / style 问题 |
| 回答看似对，但代码证据不支持 | 幻觉 / grounding 问题     |

这一步非常重要。否则你们会陷入一个误区：**看到最终回答错了，就盲目改 prompt。**
实际上很多错答案根本不是 prompt 问题，而是前面的卡片缺失、关系缺失、召回失败。

---

## 1.4 黄金数据有限时，怎么扩充评测集？

不要一上来追求大规模人工标注。建议分三类数据。

### A. 小而精的人工黄金集

比如 50～200 条。

覆盖这些问题类型：

| 问题类型   | 示例                                  |
| ------ | ----------------------------------- |
| 业务流程解释 | “订单取消后退款怎么走？”                       |
| 状态机解释  | “退款单有哪些状态？”                         |
| 异常链路解释 | “支付回调失败怎么办？”                        |
| 影响面分析  | “改这个优惠规则会影响哪些业务？”                   |
| 字段含义解释 | “refund_status 和 pay_status 有什么区别？” |
| 跨模块调用  | “售后完成后为什么会通知订单域？”                   |
| 反事实问题  | “如果用户已发货还能取消吗？”                     |
| 无答案问题  | “这个系统是否支持海外支付？”                     |

### B. 自动生成的银标集

从知识卡片自动生成问题，再让 LLM 生成参考答案，然后人工抽检。

例如每张知识卡片生成：

* 3 个事实型问题
* 3 个流程型问题
* 2 个异常型问题
* 2 个影响面问题
* 1 个“证据不足时应该拒答”的问题

注意：银标集不能直接当最终权威，只能用于回归测试和召回压力测试。

### C. 线上真实问题集

从业务同学真实提问里收集：

* 高频问题
* 低满意度问题
* 人工纠正过的问题
* 模型答非所问的问题
* 多轮追问里暴露理解不足的问题

OpenAI 的评测实践建议里也强调：要尽早、持续地做 eval；测试要贴近真实分布；记录日志；自动化评测和人工判断结合；持续维护人类反馈和自动评分的一致性。([OpenAI 开发者][6])

## 1.5 迭代优化的正确姿势：做消融实验

每次改 pipeline，不要凭感觉说“好像更好了”。你们应该固定一套 eval set，然后做版本对比。

例如：

| 实验版本         | 目的                         |
| ------------ | -------------------------- |
| Raw Code RAG | 直接对代码切 chunk 做 RAG         |
| JSON RAG     | 只用 json 描述做 RAG            |
| Card RAG     | 只用知识卡片做 RAG                |
| Wiki RAG     | 用卡片 + Wiki 结构召回            |
| Graph RAG    | 用图谱路径增强召回                  |
| Hybrid RAG   | 向量召回 + 关键词 + 图谱扩展 + rerank |

重点看：

```text
最终答案准确率
关键卡片 Hit@K
图谱路径召回率
幻觉率
业务风格得分
平均延迟
平均成本
```

最终你们会发现一个规律：

> 数据层决定上限，召回层决定能不能拿到上限，生成层决定能不能把上限表达出来。

# 2. 数据层二阶段处理合理吗？

合理，而且方向是对的。

你们的策略本质是：

```text
原始代码仓
  ↓
结构化中间表示 JSON
  ↓
业务视角知识卡片 Markdown
```

这个设计类似“编译器中间表示 IR”的思想。不要让 LLM 直接从大量原始代码生成业务卡片，因为原始代码噪声太大、上下文太散、跨文件关系难以保持。先抽成 JSON，可以把结构、调用、依赖、接口、数据表、状态流转等信息固定下来，再让 LLM 做业务语义提炼。

这也符合 Karpathy LLM Wiki 的核心思想：不是每次查询时从原始资料里重新拼答案，而是先构建一个持久、结构化、相互链接的 Markdown Wiki。([Gist][7])

## 2.1 二阶段方案的优点

### 优点一：降低 LLM 的理解负担

原始代码里有大量无关信息：

```text
import
日志
异常包装
工具函数
DTO 转换
参数校验
框架注解
空值判断
```

如果直接让模型读代码，很容易被实现细节淹没。

JSON 可以先保留结构化信息：

```json
{
  "module": "refund",
  "class": "RefundService",
  "method": "createRefund",
  "business_action": "创建退款单",
  "called_by": ["OrderCancelService.cancelOrder"],
  "calls": ["PaymentGateway.refund"],
  "reads_tables": ["order", "payment_order"],
  "writes_tables": ["refund_order"],
  "domain_events": ["RefundCreatedEvent"],
  "status_transition": [
    "INIT -> PROCESSING",
    "PROCESSING -> SUCCESS",
    "PROCESSING -> FAILED"
  ]
}
```

这样第二阶段生成知识卡片时，模型面对的是“业务骨架”，不是杂乱代码。

---

### 优点二：方便做质量控制

如果直接 code → markdown，错了很难定位。

二阶段以后可以分别检查：

```text
code → json 是否抽对？
json → markdown 是否讲对？
```

例如最终卡片漏了“退款失败补偿”，你可以追：

* JSON 里有没有 `RefundRetryJob`？
* JSON 里有没有 `FAILED -> RETRYING` 状态？
* 如果 JSON 没有，是数据抽取问题；
* 如果 JSON 有但 markdown 没写，是卡片生成 prompt 问题。

---

### 优点三：更适合增量更新

代码仓每天都变。如果直接全量重建卡片，成本高且不稳定。

二阶段后可以做：

```text
代码 diff
 ↓
影响到哪些类 / 函数 / 接口 / 表
 ↓
更新对应 JSON 节点
 ↓
只重写受影响的知识卡片
 ↓
只更新相关图谱边
```

这对企业内部知识库非常关键。

---

## 2.2 但这个方案有一个大风险：JSON 信息损失

二阶段不是天然更好。它成立的前提是：

> JSON 必须是足够完整的中间表示，而不是过早压缩后的摘要。

如果第一阶段 JSON 只写成这样：

```json
{
  "class": "RefundService",
  "description": "处理退款逻辑"
}
```

那第二阶段再强也没用，因为关键信息已经丢了。

JSON 至少要保留这些信息：

| 类别    | 建议保留字段                                  |
| ----- | --------------------------------------- |
| 代码结构  | repo、module、package、file、class、method   |
| 调用关系  | called_by、calls、入口 API、下游服务             |
| 数据关系  | 读写表、关键字段、状态枚举                           |
| 业务动作  | 这个函数在业务上做什么                             |
| 状态流转  | before_state、after_state、触发条件           |
| 分支条件  | if/else 对应的业务规则                         |
| 异常处理  | 失败、重试、补偿、幂等                             |
| 配置开关  | feature flag、灰度、租户配置                    |
| 外部依赖  | MQ、RPC、HTTP、定时任务                        |
| 权限/角色 | 谁能触发、哪些用户可见                             |
| 代码证据  | file path、symbol、line range、commit hash |
| 置信度   | 规则抽取/LLM 推断/人工确认                        |

一句话：**JSON 不要只是“代码摘要”，而要成为“业务语义 IR”。**

# 3. 数据层怎么评测、迭代、优化？

数据层评测建议分成三类：结构正确性、业务表达质量、可追溯性。

---

## 3.1 code → json 的评测

这一层尽量少依赖 LLM 主观判断，多用程序化校验。

### 结构覆盖率

检查 JSON 是否覆盖代码仓关键对象：

```text
函数覆盖率 = JSON 中出现的关键函数数 / 仓库中关键函数总数
接口覆盖率 = JSON 中出现的 API 数 / 实际 API 数
表覆盖率 = JSON 中出现的数据表数 / 实际访问表数
MQ 覆盖率 = JSON 中出现的 Topic 数 / 实际 Topic 数
配置覆盖率 = JSON 中出现的配置项数 / 实际配置项数
```

### 关系准确率

抽样检查：

```text
A calls B 是否真实存在？
A writes table_x 是否真实存在？
API_x 是否真的进入 Service_y？
状态 A -> B 是否真的由该函数触发？
```

可以让静态分析工具、AST、grep、调用图、框架路由解析共同参与，不要完全依赖 LLM。

### 关键业务规则覆盖率

比如代码里有：

```java
if (order.isShipped()) {
    throw new BizException("已发货订单不能取消");
}
```

JSON 里应该有：

```json
{
  "business_rule": "已发货订单不能取消",
  "condition": "order.status == SHIPPED",
  "source": "OrderCancelService.java:88"
}
```

如果没有，这就是数据层漏抽。

---

## 3.2 json → markdown 知识卡片的评测

知识卡片建议固定模板，不要让模型自由发挥。

### 推荐卡片结构

```markdown
# 订单取消业务卡片

## 业务问题
这张卡片解释用户取消订单时，系统如何处理订单状态、退款和后续通知。

## 适用场景
- 用户主动取消订单
- 商家超时未接单自动取消
- 风控拦截导致取消

## 主流程
1. 校验订单是否允许取消
2. 更新订单状态为已取消
3. 判断是否需要退款
4. 创建退款单
5. 调用支付渠道退款
6. 等待支付渠道回调
7. 更新退款最终状态

## 异常流程
- 如果支付渠道退款失败，会进入退款补偿任务
- 如果重复收到回调，通过退款单号做幂等处理

## 关键状态
- order_status: CREATED / PAID / CANCELLED
- refund_status: INIT / PROCESSING / SUCCESS / FAILED

## 相关代码证据
- OrderCancelService.cancelOrder
- RefundService.createRefund
- PaymentCallbackHandler.handleRefundCallback
- RefundRetryJob

## 关联卡片
- 支付退款卡片
- 退款补偿卡片
- 订单状态机卡片
```

### 卡片质量评分 Rubric

| 维度   | 评分标准             |
| ---- | ---------------- |
| 准确性  | 是否忠实于 JSON 和代码   |
| 完整性  | 是否包含主流程、异常流、边界条件 |
| 业务抽象 | 是否从函数调用提升为业务动作   |
| 可读性  | 业务同学是否能看懂        |
| 可追溯性 | 每个关键结论是否有代码证据    |
| 可召回性 | 标题、小标题、关键词是否利于检索 |
| 可链接性 | 是否指向相关卡片和业务实体    |

建议每张卡片输出一个自动评分：

```json
{
  "card_id": "order_cancel_flow",
  "accuracy": 4,
  "completeness": 3,
  "business_abstraction": 5,
  "traceability": 4,
  "retrievability": 4,
  "problems": [
    "异常流程缺少重复回调幂等说明",
    "没有关联支付退款卡片"
  ]
}
```

## 3.3 数据层的迭代优化闭环

建议你们按这个流程做：

```text
1. 固定一批核心业务域
2. 每个业务域人工维护 5～10 张标杆卡片
3. 用标杆卡片反推卡片生成 prompt 和 JSON schema
4. 批量生成卡片
5. 自动评测 + 人工抽检
6. 将错误分类
7. 更新 JSON 抽取规则 / 卡片 prompt / schema
8. 回归测试
```

错误分类可以这样做：

| 错误类型   | 修改方向                 |
| ------ | -------------------- |
| 代码关系漏抽 | 增强 AST / 调用图 / 框架解析  |
| 业务规则漏抽 | 增强 if/else、异常、状态枚举抽取 |
| 卡片太代码化 | 修改 prompt，要求“业务动作优先” |
| 卡片太空泛  | 强制引用代码证据和状态流转        |
| 卡片重复   | 增加实体归并和卡片去重          |
| 关系边错误  | 引入人工审核或高置信度规则        |
| 卡片不可召回 | 优化标题、别名、业务关键词        |

# 4. 一个比较适合你们的整体架构

建议最终做成这个闭环：

```text
代码仓库
  ↓
静态分析 / AST / 调用图 / 路由解析 / DB访问解析
  ↓
结构化 JSON IR
  ↓
JSON 质量检查
  ↓
业务知识卡片 Markdown
  ↓
卡片质量检查
  ↓
实体抽取 / 关系抽取 / 卡片链接
  ↓
LLM Wiki / Graph
  ↓
混合召回：向量 + 关键词 + 图谱路径 + rerank
  ↓
业务视角回答生成
  ↓
答案评测
  ↓
错误归因
  ↓
反向优化数据层 / 编译层 / 生成层
```

关键不是“做一个更强的 prompt”，而是做一个**可观测、可评测、可归因、可回归的代码理解系统**。

# 5. 最实用的落地建议

你们现在可以先做 5 件事。

## 第一，重构黄金数据集

不要只存黄金答案，改成：

```text
问题
黄金答案
必须命中的知识卡片
必须覆盖的业务点
必须引用的代码证据
必须经过的图谱路径
禁止出现的错误说法
业务语言风格要求
```

这是整个评测体系的地基。

---

## 第二，给每条回答保存 trace

每次回答都保存：

```text
召回了哪些卡片
走了哪些图谱边
用了哪些代码证据
最终答案包含哪些业务 claim
```

没有 trace，就无法知道错在数据层、编译层还是生成层。

---

## 第三，先评测 Top 20 核心业务问题

不要一开始铺全仓库。先选业务同学最常问、最有价值的 20 个问题。

例如：

```text
订单取消怎么走？
退款失败怎么处理？
优惠券什么时候退回？
支付回调重复怎么办？
售后单和退款单是什么关系？
某个状态为什么不能修改？
```

围绕这些问题，把三层 pipeline 打磨通。

---

## 第四，卡片必须绑定代码证据

知识卡片里每个关键业务结论都要能回到：

```text
repo
file
class
method
line range
commit hash
```

否则卡片会变成“AI 写的业务文档”，无法长期维护。

---

## 第五，建立版本对比机制

每次改动都跑同一批 eval：

```text
v1: 原始 code RAG
v2: JSON RAG
v3: Card RAG
v4: Card + Wiki
v5: Card + Wiki + Graph
```

比较：

```text
答案准确率
关键卡片召回率
关键路径召回率
业务语言得分
幻觉率
延迟
成本
```

---

## 最核心的一句话

你们真正要搭的不是“代码问答机器人”，而是一个：

> 把代码仓库编译成业务知识，再基于业务知识回答问题的代码理解系统。

所以优化重点不是单点 prompt，而是：

```text
黄金答案结构化
→ 中间过程可观测
→ 每层单独评测
→ 错误可归因
→ 版本可回归
→ 线上问题反哺知识库
```

数据层的二阶段策略是合理的，但前提是：**JSON 要足够结构化、可追溯、可增量更新；知识卡片要业务化、证据化、可召回、可链接。**
